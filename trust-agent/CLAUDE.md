```bash
npm install           # install dependencies

npm run example       # run full A2A lifecycle demo (example.ts)
npm run proxy:test    # run 4-scenario integration test (proxy-test.ts)
```

There is no test framework. `example.ts` and `proxy-test.ts` are the integration tests; run them with `tsx` via the npm scripts above. `tsconfig.json` excludes these files from the build — only `src/**/*` is compiled into `dist/`.

The package is ESM (`"type": "module"`), targets ES2022, and uses `NodeNext` module resolution. All internal imports must use `.js` extensions (TypeScript's NodeNext requirement).

## Architecture

This is `@trustagentai/a2a-core` — a cryptographic accountability layer that wraps MCP (Model Context Protocol) tool calls with tamper-evident, legally-grade receipts. It operates as a **sidecar proxy**, requiring no changes to existing agents or tools.

### The Three-Phase Protocol

Every high-stakes tool call produces three signed artifacts in sequence:

1. **IntentEnvelope** — Proxy A signs before the call: "I intend to invoke tool X with these args"
2. **AcceptanceReceipt** — Proxy B signs after validation: "I received and accepted this intent"
3. **ExecutionEnvelope** — Proxy B signs after execution: "I executed it, here is the outcome hash"

The `trace_id` links all three artifacts. Raw `args` and `outputData` are **never stored** — only their SHA-256(JCS) hashes appear in envelopes (privacy by design).

### Hash Target Rule (§4 of spec)

All envelope hashes are computed over the object **with the `signatures` field removed**. This is enforced in [src/crypto.ts](src/crypto.ts) (`computeEnvelopeHash`) and means signatures can be added/updated without changing the envelope's identity hash. Always strip `signatures` (and `entry_hash` for ledger entries) before hashing.

### Module responsibilities

| File | Role |
|------|------|
| [src/crypto.ts](src/crypto.ts) | Ed25519 key generation, signing, verification; JCS canonicalization (RFC 8785); `generateNonce()` |
| [src/envelopes.ts](src/envelopes.ts) | Builders for `IntentEnvelope`, `AcceptanceReceipt`, `ExecutionEnvelope`; dual-sign support |
| [src/ledger.ts](src/ledger.ts) | `DAGLedger`: append entries, auto-batch at `batchSize` (default 8), manual `flush()`, `getDisputePack()`, Merkle tree utilities |
| [src/nonce-registry.ts](src/nonce-registry.ts) | Anti-replay: `(did, nonce)` uniqueness with TTL + 5 s skew tolerance |
| [src/risk-budget.ts](src/risk-budget.ts) | `RiskBudgetEngine`: per-agent tool whitelist, single-action cap, rolling 24 h budget |
| [src/trust-proxy.ts](src/trust-proxy.ts) | `ProxyAGateway` (sends IntentEnvelope, awaits AcceptanceReceipt, fires ExecutionEnvelope); `ProxyBGateway` (validates TTL → nonce → signature → budget → records to DAG) |
| [src/proxy-server.ts](src/proxy-server.ts) | Node `http` server wrapping `ProxyBGateway`: `POST /accept`, `POST /executed`, `POST /flush`, `GET /history/:id`, `GET /dispute/:id` |

### DAG Ledger & Merkle Batching

`DAGLedger` stores entries in a DAG (each entry references `prev_entry_hashes`). When `pendingBatchEntries.length >= batchSize`, it auto-commits a `MerkleBatch`. Call `ledger.flush()` to commit manually before L2 anchoring. The `getDisputePack(traceId)` method returns all entries for a trace plus their Merkle inclusion proofs — this is the evidence bundle for dispute resolution.

`computeMerkleRoot` uses Bitcoin-style odd-leaf duplication. `verifyMerkleProof` re-derives the root from a leaf + proof path and checks against the stored root.

### ProxyB validation order

`ProxyBGateway.handleIntent()` enforces checks in this sequence: TTL → nonce uniqueness → Ed25519 signature → risk budget. Failure at any step returns an MCP error code (ERR_EXPIRED `-32004`, ERR_REPLAY `-32003`, ERR_INVALID_SIGNATURE `-32005`, ERR_BUDGET_EXCEEDED `-32002`). Budget `recordSpend()` is only called on `COMPLETED` execution (in `handleExecution`), not at acceptance time.

### Demo vs. production

The proxy server (`proxy-server.ts`) generates ephemeral keys at startup for demo purposes. In production, Proxy A public keys would be loaded from a DID resolver / Key Registry, and `AgentPolicy` entries would be loaded from Verifiable Credentials. The `anchored_at` field on `MerkleBatch` is a placeholder; real L2 anchoring (Base/Arbitrum) is a v0.5+ roadmap item.

## Key invariants

- `spec_version` is `"0.4"` in all current envelopes.
- `envelope_type` must be set correctly — it participates in the JCS hash.
- Nonces are 8 random bytes (hex) generated by `generateNonce()` in crypto.ts.
- Default TTL is 30 seconds (`ttlSeconds` param on envelope builders).
- `ProxyBGateway` uses two internal maps (`intentEntryHashes`, `intentMeta`) that are cleaned up after `handleExecution` completes — they are not persisted across restarts.

1. Think Before Coding
Don't assume. Don't hide confusion. Surface tradeoffs.

Before implementing:
State your assumptions explicitly. If uncertain, ask.
If multiple interpretations exist, present them - don't pick silently.
If a simpler approach exists, say so. Push back when warranted.
If something is unclear, stop. Name what's confusing. Ask.

2. Simplicity First
Minimum code that solves the problem. Nothing speculative.
No features beyond what was asked.
No abstractions for single-use code.
No "flexibility" or "configurability" that wasn't requested.
No error handling for impossible scenarios.
If you write 200 lines and it could be 50, rewrite it.
Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

3. Surgical Changes
Touch only what you must. Clean up only your own mess.

When editing existing code:
Don't "improve" adjacent code, comments, or formatting.
Don't refactor things that aren't broken.
Match existing style, even if you'd do it differently.
If you notice unrelated dead code, mention it - don't delete it.

When your changes create orphans:
Remove imports/variables/functions that YOUR changes made unused.
Don't remove pre-existing dead code unless asked.
The test: Every changed line should trace directly to the user's request.

4. Goal-Driven Execution
Define success criteria. Loop until verified.

Transform tasks into verifiable goals:
"Add validation" → "Write tests for invalid inputs, then make them pass"
"Fix the bug" → "Write a test that reproduces it, then make it pass"
"Refactor X" → "Ensure tests pass before and after"
For multi-step tasks, state a brief plan:

1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.
